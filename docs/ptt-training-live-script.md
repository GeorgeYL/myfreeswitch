# 对讲培训现场演示口播稿 + 操作清单（按分钟）

## 1. 演示目标

本次演示目标：

- 证明 4 场地 x 4 信道可独立运行，组内互通、组间隔离。
- 证明系统自动记录日志并关联录音文件，可供第三方实时读取。
- 证明机器人可在指定信道自动回复，且全组可听到。

建议时长：15 分钟（可压缩到 10 分钟）。

---

## 2. 演示角色与设备准备

角色分工：

- 讲解员 1 人：负责口播和操作台。
- 学员终端 3 人：A、B、C（手机 SIP 软终端或硬对讲终端）。

设备与软件：

- FreeSWITCH 服务端已启动。
- Demo API 服务已启动（run_demo.ps1）。
- 三台终端均在同一 WiFi，且已注册到 FreeSWITCH。
- 可访问 API 的浏览器或 PowerShell。

建议事前确认：

- FreeSWITCH 已执行 `reloadxml`。
- 终端注册状态正常（可拨号）。
- `http://127.0.0.1:8090/health` 返回 status=ok。

---

## 3. 按分钟操作清单 + 口播稿

### 第 0-1 分钟：开场与目标说明

操作：

1. 打开监控屏幕（可看到终端、API 返回、日志）。

口播（可直接照读）：

“各位好，接下来我们用 15 分钟演示培训对讲方案。核心看三件事：第一，按场地和信道分组互不干扰；第二，通话日志和录音自动留痕；第三，机器人能够在对应信道自动回复。演示结束后，大家可以按同样步骤复现。”

---

### 第 1-3 分钟：说明拨号规则

操作：

1. 在白板或屏幕展示规则：`7xy`。
2. 说明 x=场地编号（1-4），y=信道编号（1-4）。

口播：

“我们采用统一拨号规则 7xy。x 是场地，y 是信道。比如 711 代表场地 1 信道 1，712 代表场地 1 信道 2。每个信道会进入一个独立会议房间，所以组间天然隔离。”

---

### 第 3-6 分钟：同组互通验证（A/B）

操作：

1. 终端 A 拨号 `711`。
2. 终端 B 拨号 `711`。
3. A 说一句话，B 复述确认；B 再说一句，A 复述确认。

口播：

“现在 A 和 B 都进入 711，也就是同一组。请 A 先说一句测试语音。现在 B 可以清楚听到并复述。反向也一样，说明组内双向实时对讲正常。”

预期结果：

- A/B 可互相听到，语音清晰、低时延。
- 通话不中断，无明显卡顿。

---

### 第 6-8 分钟：组间隔离验证（C）

操作：

1. 终端 C 拨号 `712`。
2. 让 A/B 持续对话 10 秒。
3. 询问 C 是否听到 A/B；再让 C 说话并询问 A/B 是否听到。

口播：

“现在 C 进入 712，与 A/B 的 711 不同信道。请 A/B 继续通话。可以看到 C 听不到 711 内容，C 的说话也不会进入 711，说明组间隔离生效。”

预期结果：

- C 与 A/B 互不可听。
- 证明不同信道互不干扰。

---

### 第 8-10 分钟：机器人自动回复验证

操作（方式 A，推荐现场体验）：

1. 在 711 任意终端按键 `2`。
2. 全组确认听到机器人安全提示语音。

口播：

“现在我们触发机器人自动答疑。在 711 内按键 2，系统会匹配预设答案并在当前信道广播，组内所有成员都能听到。”

操作（方式 B，展示对接能力）：

1. 在 PowerShell 执行：
   `Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8090/api/bot/reply" -ContentType "application/json" -Body '{"site":1,"channel":1,"question":"need safety reminder"}'`

口播：

“除了按键触发，也可以由第三方分析模块通过 API 下发回复，这一步等价于‘识别到问题后自动播报答案’。”

预期结果：

- 711 内听到机器人播报。
- API 返回 room、answer_file、freeswitch_result。

---

### 第 10-12 分钟：日志与录音可追溯验证

操作：

1. 让 A/B/C 先后挂机。
2. 执行日志查询：
   `Invoke-RestMethod -Method Get -Uri "http://127.0.0.1:8090/api/logs" | ConvertTo-Json -Depth 6`
3. 选择一条 call_id，下载录音：
   `Invoke-WebRequest -Uri "http://127.0.0.1:8090/api/recordings/{call_id}" -OutFile ".\demo.wav"`

口播：

“每通电话都会自动生成日志并关联录音。这里可以看到设备号、IP、信道、起止时间、时长和录音路径。我们再按 call_id 下载录音，验证日志和录音一一对应。”

预期结果：

- 返回日志数组，字段完整。
- 录音可下载并可播放。

---

### 第 12-14 分钟：第三方接口接入说明

操作：

1. 展示接口清单：
- GET /health
- GET /api/logs
- GET /api/logs/{call_id}
- GET /api/recordings/{call_id}
- POST /api/bot/reply

口播：

“第三方系统不需要直接改 FreeSWITCH，只需调标准 HTTP 接口即可读取日志、获取录音、触发机器人回复，便于后续接入教务系统、质检平台或问答引擎。”

---

### 第 14-15 分钟：总结与收口

口播：

“本次演示完成三项验证：分组隔离、日志录音留痕、机器人自动回复。当前是可运行 Demo，若进入生产，我们会进一步增加鉴权审计、数据库归档、对象存储和 ASR/NLP/TTS 智能链路。”

---

## 4. 现场异常处理口播模板

### 场景 A：某终端未注册

口播：

“该终端暂未完成注册，我们先用已注册终端继续演示，不影响核心能力验证。”

处理动作：

1. 检查终端账号密码和服务器地址。
2. 改用备用终端继续。

### 场景 B：机器人未播报

口播：

“机器人服务已收到触发，我们马上做连通性排查并重试一次。”

处理动作：

1. 访问 /health。
2. 检查 bot_audio 文件是否存在。
3. 再次调用 /api/bot/reply。

### 场景 C：日志暂未出现

口播：

“日志在挂机后写入更完整，我们现在结束通话并刷新一次接口。”

处理动作：

1. 确认通话已挂机。
2. 重新 GET /api/logs。

---

## 5. 一页式导演提示卡（极简）

- 0:00 介绍目标。
- 1:00 讲 7xy 规则。
- 3:00 A/B 拨 711，验证组内互通。
- 6:00 C 拨 712，验证组间隔离。
- 8:00 按键 2 或 API 触发机器人。
- 10:00 挂机后查日志并下载录音。
- 12:00 介绍第三方接口。
- 14:00 总结并收口。

可选压缩版（10 分钟）：

- 合并“规则说明”和“互通验证”。
- 机器人仅演示一种触发方式。
- 日志展示只做 /api/logs，不现场下载录音。
